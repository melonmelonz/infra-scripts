The one rule that explains the whole RAM plan

GPU passthrough pins a VM's RAM for DMA, so a passthrough VM cannot balloon. Windows-main's ~138 GB only returns to the pool when that VM is powered off — not when the friend closes Moonlight, not when Windows is idle or asleep. That's why hugepages are on-demand (not boot-reserved) and why the Linux compile VM only grows past its ~16 GB floor when you deliberately shut Windows down ("Big-Linux mode").

0Pre-stage the MikroTik at your bench

Yes — you can absolutely build the VLANs now at your place, then ship the router to your friend. RouterOS config lives on the device, so a router you configure here arrives ready to drop in. This is the smart move: do the fiddly, lock-out-prone part on a bench where a mistake costs nothing.

What "pre-staging" means here

You power the RB5009 on your desk, push the full VLAN + firewall config, verify it boots clean, then box it up. At the friend's you just plug WAN + the trunk + access ports into the right jacks and it works. Nothing in the config depends on being physically at the friend's except the WAN DHCP lease, which it picks up on arrival.

The VLAN map you're staging

VLANNameSubnetMembers
10mgmt10.10.10.0/24Router, PVE host mgmt, BMC/IPMI, PBS, Linux-compile VM
20servers10.10.20.0/24Services VM (Jellyfin/DBs), DayZ VM, backup box
30gaming10.10.30.0/24Windows-main VM + friend's mini PC
40trusted10.10.40.0/24Penn's laptop, workstations, spare ports
Why the mini PC and Windows-main share VLAN 30

Same broadcast domain = Moonlight auto-discovery (mDNS) works, and the latency-critical stream is L2 hardware-switched — it never touches the router CPU. That's how you get sub-millisecond in-room streaming. Keep them together.

Bench procedure

  1. 1Update firmware first. In Winbox: /system package update then /system routerboard upgrade, reboot between. Set a strong admin password; disable telnet/ftp/www/api-ssl.
  2. 2Connect via Winbox over MAC (not IP) — it survives the IP changes you're about to cause, so you can't lock yourself out mid-config.
  3. 3Paste the full config from docs/network-topology.md — bridge, ports, VLAN membership, gateways, DHCP, NAT, firewall — but leave the very last line off.
  4. 4Verify you still have access, then enable VLAN filtering last:
# ONLY after everything else is in and you've confirmed access:
/interface bridge set bridge vlan-filtering=yes
Classic lock-out

Turning on vlan-filtering=yes with a wrong tagged/untagged map cuts your own connection. On a bench that's a 30-second power-cycle; at the friend's with the router in a closet it's a bad evening. This is the entire reason to stage at home.

Green light before you box it up

  • /interface bridge vlan print shows the tagged/untagged map you expect.
  • /ip address print shows all four gateways (…10.1, .20.1, .30.1, .40.1).
  • A laptop on ether2 gets a 10.10.30.x lease; on ether5 gets 10.10.40.x.
  • That laptop can reach the internet through the router.

Label the ports with tape (WAN / trunk / mini-PC / IPMI / backup) so the friend-side plug-in is foolproof, then ship it.

★ The IPMI lifeline — do not skip

The WRX90E-SAGE SE has an AST2600 BMC with a dedicated IPMI port. Get it on VLAN 10 (mgmt), set a password, and confirm the remote KVM (HTML5 console) actually works from another machine before you rely on it. When passthrough binds both GPUs to vfio, the host has no local video — IPMI is your only console. Verify it's populated on the SE variant specifically; that's an open item in the spec.

Green light: you can open the IPMI HTML5 console from a second machine, see the Proxmox login, and "mount ISO" works. That's your remote hands.

2Proxmox host prep (VFIO, ARC, NUT, Tailscale)

Proxmox is installed; now configure the host-level things OpenTofu can't manage. This is exactly what the ansible/ role in this repo does — one playbook does VFIO binding, ARC cap, NUT, and Tailscale.

1Discovery — collect the hardware-specific values (run on the host)

lspci -nn | grep -iE 'nvidia|audio'   # -> the PCI IDs: GPU + audio fn of EACH card
ip -br link                            # -> the NIC name (10G)
ls -l /dev/disk/by-id/                 # -> stable disk paths if needed

Each NVIDIA card shows two functions — the GPU and its HDMI-audio device. You pass both. So you'll list four IDs total (5080 GPU + 5080 audio, 3080 GPU + 3080 audio).

2Fill the repo vars

  • ansible/group_vars/all.yml → vfio_pci_ids (the four device-IDs).
  • ansible/inventory.ini → the host's mgmt IP.

3Run the play (pass secrets at runtime — never commit them)

cd ansible
ansible-playbook site.yml \
  -e "tailscale_authkey=tskey-XXXX nut_monitor_password=SOMETHING"

This sets the kernel cmdline via proxmox-boot-tool (ZFS root uses systemd-boot, not grub — important detail), binds vfio-pci, blacklists the NVIDIA drivers on the host, caps zfs_arc_max ~8 GB, installs NUT against the APC, and brings up Tailscale.

4Reboot, then verify

./scripts/verify-passthrough.sh
Do not proceed until green

Every GPU and audio function must show Kernel driver in use: vfio-pci, each card in a clean, isolated IOMMU group, ARC capped, NUT talking to the UPS, Tailscale up. If a GPU is sharing an IOMMU group with something else, fix that (ACS / slot choice) before building VMs — a dirty group means passthrough will be flaky in ways that look like random VM crashes.

Green light: both GPUs on vfio-pci in clean groups · ARC capped · NUT + Tailscale healthy.

Make vmbr0 VLAN-aware (Proxmox)

In the Proxmox UI: Datacenter → host → System → Network → edit vmbr0 (the bridge on the 10G NIC) → tick VLAN aware. Then each VM's vNIC gets a VLAN Tag set to its segment (Windows = 30, Services/DayZ = 20, Linux-compile = 10). The host's own mgmt IP sits on VLAN 10.

Put the host mgmt + IPMI on Tailscale

Install Tailscale on the host (the Ansible role did this) and confirm you can reach both the Proxmox UI and the IPMI console over Tailscale from outside. That's your out-of-band recovery path for a box you don't physically sit next to.

Green light: mini PC on ether2 pulls a 10.10.30.x lease · host reachable on VLAN 10 · Tailscale reaches host + IPMI · internet works from each VLAN.

4★ The Windows-main VM (the friend's machine)

This is the one your friend actually uses — Trion, point-cloud work, and gaming, all streamed to his in-room mini PC. It's the first VM you build and the most detailed section here. Take your time; get this rock-solid before touching the others.

★ Which Windows does he need? — Windows 11 Pro (not Home)

This isn't a preference, it's a hard requirement: Windows 11 Home caps at 128 GB of RAM. This VM gets ~138 GB. Home would simply refuse to address the extra memory — the whole point of the build (holding a 120 GB+ point cloud in RAM) breaks. Pro supports up to 2 TB, plus it gives you RDP host and cleaner headless behavior. Get a retail Windows 11 Pro license. (Pro for Workstations isn't needed — that's for ReFS / 4-socket / >2 TB; plain Pro covers you.)

2Install Windows

  • Attach two CD drives: the Windows 11 Pro ISO and the virtio-win ISO.
  • When the installer shows no disks, click Load driver → browse the virtio ISO → load the vioscsi (storage) driver. The disk appears.
  • Also load the NetKVM (network) driver so networking works post-install.
  • Install Windows 11 Pro, enter the key.

3Add the 5080 passthrough

  • VM Hardware → Add → PCI Device: the 5080 GPU, tick All Functions (grabs the HDMI-audio fn too), PCI-Express, and Primary GPU.
  • Boot. Install the current NVIDIA driver inside Windows.
Blackwell + consumer passthrough

Modern NVIDIA drivers officially support consumer-card passthrough — no "Code 43" hack needed. Keep the KVM hidden flag handy only if a driver ever misbehaves. The 5080 is new silicon; the recent Proxmox kernel (Phase 1) is what gives it clean PCIe enumeration. On a VM reboot it usually FLR-resets fine; worst case a host reboot clears a stuck GPU, recoverable over IPMI.

4Make it a capturable, always-ready desktop

  • Dummy HDMI plug into the 5080 (preferred over a virtual display driver) so Windows always has a display to render/capture.
  • Enable auto-login so a power-cycle boots straight to a ready desktop — the friend never sees a lock screen.
  • Set display resolution to the target (4K60 or 1080p120).

5Install Trion (point-cloud)

  • Install Trion + the point-cloud toolchain.
  • If Trion uses a USB license dongle, pass the physical USB device through to this VM (Hardware → Add → USB Device).
★ Highest-risk open item — verify early

Confirm Trion actually runs under virtualization and that its dongle (if any) passes through cleanly. The spec flags this as the #1 risk. Test it the moment Windows + GPU are up — before you've built everything else on the assumption that it works.

6Sunshine + Moonlight (the in-room stream)

  • Install Sunshine on Windows-main — the 5080's NVENC makes encode essentially free.
  • On the friend's mini PC (wired to ether2, VLAN 30): install Moonlight, set it to auto-launch fullscreen on boot.
  • Because both are on VLAN 30, Moonlight auto-discovers the host; pair them (PIN).
  • Target 4K60 or 1080p120, sub-10 ms added latency on the wired LAN.
★ Green light for Phase 4

Friend sits at the mini PC, it boots straight into a fullscreen Moonlight session, the Windows desktop appears at target res/latency, Trion launches, and a real point-cloud job runs. That's the whole reason for the build — get a clean pass here before moving on.

5The other three VMs

With Windows solid, build these via OpenTofu (opentofu/, bpg/proxmox) — a manual first pass in the UI is fine too. Each is lighter than the Windows VM.

Services VM (Linux) — Jellyfin + thin DBs

  • Linux, ~16 GB (locked, since the 3080 is passed here), ~2 cores.
  • Pass the RTX 3080 + its audio fn through; Jellyfin uses NVENC for transcoding.
  • vNIC on VLAN 20. Expose only via Tailscale — never WAN.

DayZ VM (Windows) — Deer Isle + Expansion

  • Small Windows VM, no GPU, ~8 GB, 2–3 cores (DayZ is largely single-thread-bound — favors clock). vNIC on VLAN 20.
  • SteamCMD dedicated server + CF (Community Framework) + DayZ-Expansion-Core / -Vehicles / -Market + Deer Isle, Expansion mission adapted to Deer Isle.
  • Required features: Deer Isle map · fly-in safezones (Expansion zone-based, position+radius, traders inside — not teleport) · helicopters (Expansion-Vehicles).
  • Outside players reach it via Tailscale, or the tightly-scoped WAN dst-nat (UDP 2302–2306 + 27016) already in the router config — never raw-exposed beyond those ports.

Linux compile VM (NixOS) — your remote kernel box

  • NixOS (decided), no GPU, ballooning ON (floor 16 / max 150 GB), 4–6 cores pinned. vNIC on VLAN 10.
  • Pinned RFL toolchain via a flake (exact rustc/bindgen/LLVM) — reproducible, atomic rollback so a remote box is very hard to brick.
  • SSH/mosh in over Tailscale. The 50 Mbps uplink is a non-issue for compiles.
Big-Linux mode (how you reclaim RAM for huge builds)

Default is co-resident: Windows up 24/7, Linux at its ~16 GB floor — fine for most builds since -j parallelism is CPU-bound. For a giant tmpfs build: over SSH/IPMI confirm no active Sunshine session → qm shutdown <winvm> → run the job (Linux balloons up automatically) → qm start <winvm> to restore the friend's machine before he needs it. The friend never manages any of this.

6Resilience & backups

  • NUT against the APC SMT1500C (Ansible configured it) — and actually test a simulated power-loss shutdown. Mandatory for an unattended remote box. The 1000 W UPS is graceful-shutdown headroom, not ride-through.
  • Proxmox Backup Server on the 64 GB / RAIDZ1 box — kept on the local LAN so bulk backups run at 10G speed (a full 120 GB over the 50 Mbps uplink is ~5.5 h — impractical). Run a first VM backup.
  • Offsite via physical cold-drive rotation: zfs send | zstd to a bare drive in a USB dock, stored cold. Only small incrementals cross the WAN. That's your 3-2-1.
Storage discipline

The friend's real project files and point clouds live on the backed-up ZFS mirror, never on the 4 TB scratch drive (no redundancy, transient only). Make this a habit from day one.
